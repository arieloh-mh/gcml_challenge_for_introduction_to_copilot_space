## Purpose
Provide a structured onboarding experience for new team members joining OctoAcme projects. This checklist ensures consistent knowledge transfer and helps new members become productive quickly.

## When to Use
- When a new team member joins an ongoing project
- At project kickoff when assembling a new team
- When transitioning team members between projects

## General Onboarding (All Roles)

### Access & Setup
- [ ] GitHub repository access granted
- [ ] Project management tools access (boards, tracking systems)
- [ ] Communication channels added (Slack, Teams, email lists)
- [ ] Development environment setup (if applicable)
- [ ] Credentials and service accounts provisioned

### Project Context
- [ ] Review Project Charter / One-pager
- [ ] Understand problem statement and success metrics
- [ ] Review roadmap and current sprint/iteration goals
- [ ] Meet key stakeholders and team members
- [ ] Understand escalation paths and decision-making process

### Process & Documentation
- [ ] Read relevant process docs in `docs/` folder
- [ ] Understand project lifecycle and current phase
- [ ] Review communication cadence and meeting schedule
- [ ] Familiarize with definition of done and acceptance criteria
- [ ] Locate risk register and status reporting artifacts

## Role-Specific Onboarding

### Developers
- [ ] Review architecture and technical design docs
- [ ] Set up local development environment
- [ ] Run test suite successfully
- [ ] Understand branching strategy and PR process
- [ ] Shadow a code review session
- [ ] Pair with team member on a small task

### Product Managers
- [ ] Review product vision and strategy
- [ ] Understand customer segments and personas
- [ ] Access analytics and metrics dashboards
- [ ] Review backlog prioritization criteria
- [ ] Meet with key customers or stakeholders
- [ ] Review competitive landscape and market context

### Project Managers
- [ ] Review project plan, timeline, and milestones
- [ ] Understand budget and resource constraints
- [ ] Access risk register and dependencies log
- [ ] Review reporting cadence and stakeholder list
- [ ] Understand escalation procedures
- [ ] Shadow team sync meetings

### QA/Testing
- [ ] Review test strategy and test plans
- [ ] Set up test environments
- [ ] Access bug tracking system
- [ ] Understand automation framework (if applicable)
- [ ] Review acceptance criteria guidelines
- [ ] Shadow testing workflow

### UX Designer
- [ ] Review design system and style guides
- [ ] Access design tools (Figma, Sketch, etc.)
- [ ] Understand user research findings
- [ ] Review existing user flows and wireframes
- [ ] Meet with Product Manager to understand user needs
- [ ] Review accessibility standards and requirements

### DevOps Engineer
- [ ] Review infrastructure architecture
- [ ] Access CI/CD pipelines and deployment tools
- [ ] Understand monitoring and alerting systems
- [ ] Review incident response procedures
- [ ] Access infrastructure as code repositories
- [ ] Shadow a deployment or release process

### Security Specialist
- [ ] Review security policies and compliance requirements
- [ ] Access security scanning and audit tools
- [ ] Understand threat model for the project
- [ ] Review previous security assessments
- [ ] Understand incident response procedures
- [ ] Review secure coding guidelines

### Business Analyst
- [ ] Review business requirements and use cases
- [ ] Understand business processes and workflows
- [ ] Meet with key business stakeholders
- [ ] Access requirements documentation tools
- [ ] Review data models and business rules
- [ ] Understand domain-specific terminology

## First Week Goals

### Day 1
- Complete access and setup checklist
- Attend team introduction meeting
- Review project documentation

### Day 2-3
- Shadow team meetings (standup, planning)
- Complete role-specific onboarding items
- Ask questions and clarify understanding

### Day 4-5
- Take on a small, well-defined task
- Participate in code review or design review
- Provide feedback on onboarding experience

## Onboarding Buddy System
- Assign an onboarding buddy for each new team member
- Buddy should be available for questions during first 2 weeks
- Schedule regular check-ins during first week
- Buddy helps with cultural norms and informal knowledge

## Feedback & Continuous Improvement
- Collect feedback after first week and first month
- Update onboarding checklist based on learnings
- Share improvements with other teams
- Celebrate successful onboarding milestones
